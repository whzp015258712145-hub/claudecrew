# Spec 格式参考

claudecrew SPEC 协议中每种文件类型的模板和示例。

---

## REQUEST.md

用户的原始请求，原样保存。只写一次，永不修改。

```markdown
# 需求 — <一行摘要>

## 原始需求
<用户的精确请求，或最忠实的转述。>

## 澄清记录
- 问：<你提的问题> → 答：<用户的回答>
- ...

## 验收标准
- [ ] <可衡量的标准>
- [ ] <可衡量的标准>

## 约束条件
- 技术栈：<语言、框架>
- 不可触碰：<文件/目录>
- 风格：<代码风格要求>

## 完成定义
<"此需求已完成"的具体含义。>
```

---

## DISPATCH.md

分解方案和执行日志。

```markdown
# 派遣计划 — <日期> <短标识>

## 分解理由
<为什么这样拆分？独立的工作流是什么？如何确保工人之间零耦合？>

## Agent 分配
| Agent ID | 类型   | Spec 文件                   | 描述              |
|----------|--------|-----------------------------|-------------------|
| W01      | worker | tasks/worker-01_<slug>.md   | <一行摘要>         |
| W02      | worker | tasks/worker-02_<slug>.md   | <一行摘要>         |
| R01      | review | tasks/review-01_<slug>.md   | 审查 W01 + 代码审计 |
| R02      | review | tasks/review-02_<slug>.md   | 审查 W02 + 代码审计 |
| CA       | chief  | tasks/chief-auditor_<slug>.md | 跨模块集成审计   |

## 执行日志
| 时间   | 操作                          | 状态 |
|--------|-------------------------------|------|
| 14:00  | 启动 W01、W02                  | ✓    |
| 14:15  | 收到 W01 结果                  | ✓    |
| 14:18  | 收到 W02 结果                  | ✓    |
| 14:20  | 启动 R01、R02                  | ✓    |
| 14:25  | R01 裁决：PASS                 | ✓    |
| 14:26  | R02 裁决：NEEDS_FIX            | ⚠    |
| 14:30  | 为 R02 派遣 W03 修复           | ✓    |
| 14:40  | 收到 W03 结果                  | ✓    |
| 14:42  | R03 重新审查：PASS             | ✓    |
| 14:45  | 交付给用户                     | ✓    |

## 循环总结
- 启动工人：2 (W01、W02)
- 修复循环：1 (W02 → R02 → W03 → R03)
- 最终裁决：全部 PASS
```

## Worker Spec (tasks/worker-XX_<slug>.md)

```markdown
# Worker 任务：<标题>

## 你的角色
你是一个 <角色，例如"后端开发者" | "前端开发者" | "数据工程师">。
你编写生产级质量的代码。你严格遵循指令。

## 背景
<2-3 段背景信息。这是什么项目？代码库长什么样？哪些相关文件已存在？
代码库遵循什么模式？使用什么技术栈？包含 Worker 应该知道的文件路径。>

## 任务
### 步骤 1：<先做什么>
<具体指令。使用祈使语气。提及确切的文件路径。>

### 步骤 2：<接下来做什么>
<...>

### 步骤 N：<最终步骤>
<...>

## 约束
- 不得修改范围之外的文件：<明确的允许列表>
- 遵循现有代码模式：<参考文件>
- 仅使用：<允许的库/模式>
- <任何其他硬性约束>

## 预期产出
### 创建的文件：
- `<path/to/file1>` — <内容说明>
- `<path/to/file2>` — <内容说明>

### 修改的文件：
- `<path/to/file3>` — <修改内容>

### 结果摘要：
将完成报告写入：
`SPEC/<request-dir>/results/result-<agent-id>.md`

### 🔴 完成信号（强制最后一步）：
**所有工作（编写、测试、修改、自检）完成后**，创建空文件：
`SPEC/<request-dir>/results/<agent-id>已完成.md`
⚠️ 这是母 Agent 判断你完成的唯一信号。不要提前创建。
创建 已完成.md 后不再修改任何交付物。

## 验收标准（供 Reviewer 使用）
- [ ] <Reviewer 要检查的标准>
- [ ] <Reviewer 要检查的标准>
```

## Reviewer Spec (tasks/review-XX_<slug>.md)

```markdown
# 审查任务：<标题>

## 你的角色
你是一名代码审查员。你严格、彻底、有建设性。
你验证交付物是否精确符合 spec，同时审计代码中的 bug、逻辑错误和质量问题。

## 审查对象
- **Worker spec：** `SPEC/<request-dir>/tasks/worker-XX_<slug>.md`
- **Worker 结果：** `SPEC/<request-dir>/results/result-worker-XX.md`
- **交付物文件：** <Worker 应该创建/修改的文件列表>

## 审查指令
1. 通读 Worker spec——理解任务要求
2. 阅读 Worker 的结果摘要
3. 检查每个交付物文件
4. 逐条核对验收标准
5. **Bug 审计：** 扫描逻辑错误、遗漏的边界情况、null/undefined 处理、
   竞态条件、安全漏洞、资源泄漏、类型错误
6. 检查：正确性、完整性、代码质量、spec 合规性
7. 给出裁决

## 裁决选项
- **PASS** — 所有标准满足，代码达到生产级质量，未发现 bug
- **NEEDS_FIX** — 发现具体的、可修复的问题（逐一列出 file:line）
- **FAIL** — 方法根本上有问题，需要完全重做

## 预期产出
将审查报告写入：
`SPEC/<request-dir>/results/result-review-<agent-id>.md`

使用此格式：
```markdown
# 审查 — <worker-agent-id>

## 裁决：<PASS | NEEDS_FIX | FAIL>

## 验收标准检查
- [x] 标准 1 — <备注>
- [ ] 标准 2 — <有什么问题，具体说明>

## Bug 审计
| 严重程度 | 文件:行号 | 问题 | 修复建议 |
|----------|-----------|------|----------|
| 高       | ...       | ...  | ...      |
| 中       | ...       | ...  | ...      |
| 低       | ...       | ...  | ...      |

## 发现的问题
### 问题 1：<标题>
- **文件：** `<路径>:<行号>`
- **问题：** <清晰描述>
- **修复：** <可操作的建议>

### 问题 2：<标题>
- ...

## 总结
<1-2 句总体评估>
```
```

## Chief Auditor Spec (tasks/chief-auditor_<slug>.md)

Chief Auditor（审计总管）仅在所有单独审查都 PASS 后才派遣。
此 Agent 看到全貌并捕获集成问题。

```markdown
# Chief Auditor：跨模块集成审计

## 你的角色
你是审计总管（Chief Auditor）。每个单独模块都已经通过了自身的审查。
你的任务是检查所有模块在一起的情况，捕获只有在组件交互时才显现的问题。
你是交付前的最后一道防线。

## 审计对象
### Worker Spec（全部阅读）：
<列出每个 Worker spec 路径>

### Worker 结果（全部阅读）：
<列出每个 Worker 结果路径>

### 单独审查结果（全部阅读）：
<列出每个审查结果路径>

### 完整文件清单（所有创建或修改的文件）：
<所有 Worker 涉及的全部文件的完整列表>

## 审计指令

### 1. 接口兼容性检查
对于每对有交互的模块，验证：
- 函数/方法签名在调用处是否匹配
- 导入路径是否正确且可解析
- 跨边界传递的类型/结构是否兼容
- 一个模块抛出的错误类型是否被另一个模块正确捕获

### 2. 数据流完整性检查
追踪数据从入口到出口跨模块的流向：
- 数据在每个边界处是否正确转换？
- 是否存在数据丢失、截断或类型强制转换问题？
- null/undefined 值是否安全传播？

### 3. 耦合评估
- 模块是否按派遣计划的预期正确解耦？
- 有无意外的紧耦合（例如共享可变状态、循环导入）？
- 模块是否有干净、最小的接口，还是深入到了彼此的内部？

### 4. 跨模块 Bug 扫描
- 只有在模块交互时才显现的逻辑错误
- 模块间的竞态条件（例如一个模块写文件，另一个太早读取）
- 重复或冲突的实现（两个 Worker 用不同方式解决了同一问题）

### 5. 全局一致性检查
- 命名约定在所有模块中是否一致？
- 错误处理模式是否统一？
- 导入风格是否一致？
- 代码格式是否一致？

## 裁决选项
- **PASS** — 所有模块干净集成，未发现跨模块问题
- **NEEDS_FIX** — 发现具体的跨模块问题（逐一列出受影响的模块）
- **FAIL** — 分解有根本性缺陷，模块耦合太紧，无法增量修复

## 预期产出
将裁决写入：
`SPEC/<request-dir>/results/result-chief-auditor.md`

使用此格式：
```markdown
# Chief Auditor 裁决 — <request-slug>

## 裁决：<PASS | NEEDS_FIX | FAIL>

## 模块清单
| 模块 | Worker | Reviewer | 单独裁决 | 文件 |
|------|--------|----------|----------|------|
| ...  | W01    | R01      | PASS     | ...  |

## 跨模块分析

### 接口兼容性
| 来源模块 | 目标模块 | 接口点 | 状态 | 备注 |
|----------|----------|--------|------|------|
| ...      | ...      | ...    | ✓/⚠/✗ | ...  |

### 数据流
<描述数据流追踪和发现的任何问题>

### 耦合质量
<评估模块是否按预期正确解耦>

## 发现的问题
### 跨模块问题 1：<标题>
- **受影响的模块：** <列表>
- **严重程度：** 高 / 中 / 低
- **问题：** <清晰描述>
- **修复：** <可操作的建议>

## 总结
<整体集成质量评估。此交付物是否可以交给用户？>
```

## Fix Spec (tasks/fix-XX_<slug>.md)

当 Reviewer 返回 NEEDS_FIX 时，编写精确针对每个问题的修复 spec。
修复 spec 引用了审查报告，所以修复 Worker 知道要修复什么。

```markdown
# 修复任务：<标题>

## 你的角色
你是一个 <角色>。你修复代码审查中发现的问题。

## 背景
原始 Worker 任务：`SPEC/<request-dir>/tasks/worker-XX_<slug>.md`
发现问题所在的审查报告：`SPEC/<request-dir>/results/result-review-XX.md`

## 需要修复的问题
### 修复 1：<审查报告中的问题标题>
- **文件：** `<路径>`
- **要改什么：** <具体指令>
- **审查参考：** 见审查文件中的问题 1

### 修复 2：<审查报告中的问题标题>
- ...

## 约束
- 仅修复上面列出的问题 — 不得重构无关代码
- 除非审查明确要求，否则不得更改公开 API/接口

## 预期产出
修复所有问题后，将完成报告写入：
`SPEC/<request-dir>/results/result-<agent-id>.md`

### 🔴 完成信号（强制最后一步）：
**所有修复工作完成后**，创建空文件：
`SPEC/<request-dir>/results/<agent-id>已完成.md`
创建 已完成.md 后不再修改任何文件。
```
